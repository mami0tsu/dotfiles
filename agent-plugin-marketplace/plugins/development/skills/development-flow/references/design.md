# Design Flow

設計、方針決め、チケット化、実装ブリーフ作成の手順。
このフローでは `dig` skill と `difit-review` skill を必ず使う。

## 開始条件

設計が必要な依頼では、まず関連するコード、既存ドキュメント、Issue、Discussion、PR を確認する。
調べれば分かることをユーザーに聞かない。

関連情報を確認したら `dig` skill を使い、共通理解に達するまで一度に一つずつ質問する。
深掘りが終わるまで成果物を確定しない。

## 変更規模

成果物を書く前に、変更規模を保守的に分類する。

- small: 局所的で、既存の設計判断を変えず、失敗時の影響が小さい
- standard: 複数責務や複数システムにまたがる、または実装判断が残りやすい
- high-risk: データ、認証、課金、外部 API、移行、運用、セキュリティ、後戻りしにくい変更を含む

迷う場合は一段階重い分類にする。

## 成果物の選び方

成果物は目的に応じて `assets/` から選ぶ。

- 汎用の設計判断を残す: [../assets/design-doc.md](../assets/design-doc.md)
- クラウドインフラ、SLO/SLI、アラート、運用変更の設計判断を残す: [../assets/design-doc-sre.md](../assets/design-doc-sre.md)
- アプリケーション開発の設計判断を残す: [../assets/design-doc-app.md](../assets/design-doc-app.md)
- タスク管理システムや GitHub Issue に転記する: [../assets/ticket.md](../assets/ticket.md)
- 実装担当への入力を短くまとめる: [../assets/implementation-brief.md](../assets/implementation-brief.md)

必要なら複数作成する。
標準以上の変更では、少なくとも設計ドキュメントを作る。
チケットは軽量な作業管理用として扱い、詳細な設計判断が必要な場合は設計ドキュメントを参照させる。
テンプレート内の不要なセクションは削除する。

## 必須内容

small では次を必須にする。

- 背景
- 決定事項
- 実装範囲
- レビュー観点

standard と high-risk では、設計ドキュメントまたは実装ブリーフに次も必須にする。

- 非目標
- 代替案
- 未決事項
- リスク
- テスト方針
- ロールバック方針
- 影響範囲

未決事項がある場合は、実装に影響するかどうかを明記する。
実装に影響する未決事項が残る場合は、開発フローへ進めない。

## 成果物レビュー

設計成果物を作成または更新したら、commit や実装へ進む前に `difit-review` skill を使う。
設計意図、決めたトレードオフ、迷った点、特に確認してほしい設計判断を `--comment` で添える。

推奨対象:

- 未 commit の設計成果物: `difit . --include-untracked`
- staging 済みの設計成果物: `difit staged`

レビューコメントが返った場合は対応する。
レビューコメントに返信できる場合は、対応内容や判断を該当コメントへ返信する。
返信できない場合は、対応内容を最終報告に明記する。
ブラウザで開いた `difit` をユーザーが閉じるまで待つ。
レビューコメントが 0 件で返るまで、対応と `difit-review` skill によるレビューを繰り返す。
設計の前提や決定が変わった場合は、必要に応じて `dig` skill に戻る。

## 完了条件

設計フローは次を満たしたときだけ完了する。

- `dig` skill による深掘りが完了している
- 適切なテンプレートで成果物が作成または更新されている
- 変更規模に応じた必須内容が埋まっている
- 実装に影響する未決事項が残っていない
- `difit-review` skill による人間レビューが完了している
