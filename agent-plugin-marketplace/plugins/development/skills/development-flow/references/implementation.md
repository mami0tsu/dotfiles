# Implementation Flow

レビュー済みの設計成果物、チケット、実装ブリーフを入力にして開発する手順。
このフローでは原則として `dig` skill を省き、入力成果物に沿って実装する。

## 開始条件

実装前に、入力成果物を確認する。
入力成果物がない場合は実装を始めず、設計フローへ戻る。
チケットが軽量テンプレートの場合は、`参考情報` にある設計ドキュメントや実装ブリーフも入力成果物として確認する。

small 変更では、入力成果物に次が含まれていれば開始できる。

- 決定事項
- 実装範囲
- レビュー観点

standard または high-risk 変更では、チケット単体では開始しない。
設計ドキュメントまたは実装ブリーフに上記が含まれ、さらに次を満たす場合だけ開始できる。

- 未決事項が空である
- または、未決事項が実装に影響しないと明記されている

入力成果物が不足している場合は、不足項目を示して設計フローへ戻る。
この不足確認を `dig` skill の代替にしない。

## 実装前確認

実装前に次を確認する。

- 入力成果物の ticket-id
- branch type と short-description
- 変更対象のファイル、モジュール、外部連携
- テスト方針
- レビュー観点

branch や worktree が必要な場合は `git` skill を使う。
Issue、Discussion、PR、review comment を参照する必要がある場合は `github-operation` skill を使う。

## 実装ルール

入力成果物の範囲を超える変更をしない。
実装中に新しい設計判断が必要になった場合は、次のどちらかに分類する。

- 入力成果物の範囲内で自明に解ける実装詳細: 実装し、最終報告に記録する
- 設計成果物の決定事項、非目標、未決事項、リスクを変える判断: 実装を止め、設計フローへ戻る

標準以上の変更で実装に影響する未決事項を見つけた場合は、`dig` skill をその場で簡略利用せず、設計フローへ戻る。

## 実装後レビュー

実装を完了したら、commit、push、Draft PR 作成の前に `difit-review` skill を使う。
実装意図、設計との対応、トレードオフ、迷った点、重点的に確認してほしい箇所を `--comment` で添える。

推奨対象:

- 未 commit の実装変更: `difit . --include-untracked`
- staging 済みの実装変更: `difit staged`
- 直近 commit の確認: `difit`

レビューコメントが返った場合は対応する。
レビューコメントに返信できる場合は、対応内容や判断を該当コメントへ返信する。
返信できない場合は、対応内容を最終報告に明記する。
対応後に差分が変わった場合は、必要に応じて再度 `difit-review` skill を使う。
ブラウザで開いた `difit` をユーザーが閉じるまで待つ。
レビューコメントが 0 件で返るまで、対応と `difit-review` skill によるレビューを繰り返す。

## 完了条件

実装フローは次を満たしたときだけ完了する。

- レビュー済みの設計成果物、チケット、または実装ブリーフを入力にしている
- 入力成果物の範囲内で実装している
- テストまたは妥当な検証を実行している
- `difit-review` skill による人間レビューが完了している
- レビューコメントがあれば対応済みである
